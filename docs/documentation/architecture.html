    Architecture¶

    SpendScan is local-first: the OCR model runs on the user's own GPU, the database runs in a local Docker container, and only the data-extraction step calls an external service (Google Gemini). The whole application runs as one process - FastAPI serves the API, the static frontend, and drives the OCR and database clients.

    The flow of one receipt¶

    A receipt travels through four stages: OCR -> extraction -> persistence -> analytics.

                +---------------------------------------------+
  upload    | FastAPI (lifespan: preload OcrService)      |
  ------->  |   POST /api/v1/receipts                     |
            |     |-- OcrService.recognize(page_1..N)     |
            |     |     +-- llama-server (PaddleOCR-VL)   |
            |     |-- GeminiReceiptClient.analyze(...)    |
            |     |-- ReceiptRepository.save(...)         |
            |     +-- return ReceiptAnalysisResult        |
            +---------------------------------------------+
                    |                            |
                    v                            v
              workspace/uploads/         PostgreSQL
              (originals + pages)        (users, receipts, items, raw_ocr)
    1. OCR. The user uploads one or more photos that together make up one receipt (a multi-page receipt is a single upload). Each page is sent to a local llama-server running PaddleOCR-VL 1.5 (Q8 GGUF), which returns the raw text (~2 s per page on a GPU). This is handled by the OCR subpackage.
    2. Extraction. The raw OCR text and the image are sent to Google Gemini, which returns structured data: merchant, items, prices, total, discount and category. The model chain falls back from Flash to Flash Lite. This is the LLM subpackage.
    3. Persistence. The structured result is validated and stored in PostgreSQL; the original images are written to workspace/uploads/ under a per-user folder. This is the database subpackage plus the pipeline that ties the steps together.
    4. Analytics. On demand, the analysis subpackage aggregates the stored receipts into dashboard data: totals, categories, top shops and a calendar view.

    Responsibilities by subpackage¶

    Subpackage Responsibility Reference
    api The FastAPI app, the HTTP routes, request/response schemas, and dependency wiring. API layer
    ocr Recognizing text from images via PaddleOCR-VL, and managing the llama.cpp runtime (binary resolution, server lifecycle, client). OCR
    llm Building the prompt, calling Gemini, and validating the returned JSON. LLM
    db + models The database engine, the SQLModel ORM models, and the repositories (receipts, users, folders). Database
    pipeline Orchestrating OCR -> LLM -> persistence for an upload. Pipeline
    analysis Aggregating receipts into dashboard analytics. Analysis
    auth Password hashing, JWT tokens, session cookies, and the dependency that protects endpoints. Authentication
    config + errors Settings loaded from environment variables, and shared error types. Configuration & errors

    Why the frontend is served by FastAPI¶

    FastAPI mounts frontend/public/ as static files under /. This keeps everything on a single origin, so the API (/api/v1/...) and the static assets share a host - no CORS, and simple, secure cookies. It also means one process (uvicorn) brings up the frontend, the API, the OCR client and the database client together. See the Frontend reference for the UI side.

    Authentication model¶

    The session is a JWT stored in an HttpOnly cookie (SameSite=Lax), so browser JavaScript cannot read it. For requests that change data, the frontend also sends a separate CSRF token that the server checks against a cookie. Passwords are hashed with bcrypt. Every receipt read or write is tied to the logged-in user's id, so accounts are isolated. The details are in the Authentication reference.
